## 1. はじめに

### 1.1 課題

既存の半導体サプライチェーンマップ——業界団体の図表、研究機関のチャート、静的なインフォグラフィック——は**構造**を示すことには長けている。どの企業がチェーンのどの部分を占め、どの国がどの能力を有するか。しかし示せないのが**動き**である。新たな輸出規制、地政学的事象、あるいは一企業の生産能力の決定が生じたとき、どのノードが新たに露出し、その影響がどこまで及び、次に何を注視すべきかである。

半導体産業は極めて集中度が高い。オランダの一企業(ASML)がEUVリソグラフィ装置のほぼ全てを製造する。一つの島(台湾)が最先端ロジックの大半を製造する。韓国の二社がHBMメモリを支配する。この集中性ゆえに、政策・地政学・企業固有のショックは予測可能ではあるが自明ではない形でチェーンを伝播する。

### 1.2 SSCIMとは

SSCIMは以下を統合したライブ・インテリジェンス層である:

- **世界地図**(実際のOpenStreetMap地理データ)——国別に半導体能力の所在を表示
- **産業フローグラフ**——研究/IPからエンドマーケットまでの生産パイプライン全体
- **企業レイヤー**——100社を超える企業の生産シェア、顧客関係、株主構成
- **伝播エンジン**——事象・仮説シナリオ・単一企業の中断がグラフ上をどう波及するかをモデル化
- **インテリジェンスパネル**——何が変化し、なぜ重要で、次に何を注視すべきかを説明
- **ブリーフィング生成機能**——ライブなモデル状態から日次のインテリジェンス要約を作成

一文でまとめると: *チェーンがどこにあるかだけでなく、何が変化し、どのノードが影響を受け、次に何が起こり得るかを示すライブ半導体サプライチェーン・インテリジェンスマップ*。

### 1.3 想定ユーザー

政策・地政学・技術動向が半導体企業に与える影響を追う投資家・市場ウォッチャー、輸出規制とサプライチェーン安全保障を研究する政策・地政学リスク研究者、業界の構造を素早く把握したい学生・アナリスト、エクスポージャーとチョークポイントを追跡する企業戦略チーム、この分野を担当するジャーナリスト。商業仮説はこのリストが示す中間層——エンタープライズ向けデータベンダーの価格を正当化できないが、無料の公開マップ以上を必要とする買い手——を狙う。

### 1.4 関連プロダクト: GP News

SSCIMのエンジンは、人手を介さずに日次の半導体インテリジェンス・ブリーフィング(**GP News**)を生成できる。アプリ内の「GP Briefing」機能は、変化の概要・最も衝撃を受けたノード・企業エクスポージャーの上位・国別リスクボード・注視事項を、ライブなモデル状態から直接作成する。推奨されるゴートゥーマーケットの順序は、まずブリーフィングを先行させる——摩擦が少なく需要シグナルを早く得られるため——その後にプレミアム階層としてインタラクティブなダッシュボードを続ける形である。

## 2. プロダクトアーキテクチャ

SSCIMは**一つの計算エンジンの上に構築された三つの同期レイヤー**として設計されている。いずれかのレイヤーで要素を選択すると、他の二層の対応する要素がハイライトされる。

```
┌─────────────────────────────────────────────────────────────┐
│  LAYER 1: WORLD MAP          LAYER 2: INDUSTRY FLOW          │
│  (OpenStreetMap / Leaflet)   (24-stage dependency DAG)       │
│  16 countries, risk-colored  Value-weighted edges,           │
│  derived country links       importance-sized nodes          │
├─────────────────────────────────────────────────────────────┤
│  LAYER 3: INTELLIGENCE PANEL                                 │
│  Event feed · Company impact rank · Movers · Capital rank    │
│  Detail view: score breakdown, spread trees, sources          │
└─────────────────────────────────────────────────────────────┘
                          │
                          ▼
              ┌───────────────────────┐
              │  PROPAGATION ENGINE    │
              │  (one code path, three │
              │   uses: events,        │
              │   scenarios, company   │
              │   disruption)          │
              └───────────────────────┘
```

中心的な設計原則——**「一つのエンジン、三つの用途」**——は方法論的一貫性の主張である。ライブ事象、仮説シナリオ、企業中断シミュレーションは、それぞれ別の数式を持つ三つの別個の機能ではない。同一のショック伝播関数への三つの異なる*入力*である。これにより、三つのブラックボックスを一つの屋根の下に並べるのではなく、説明可能性について説得力ある主張ができる。

## 3. データモデル

### 3.1 ステージ(24ノード)

サプライチェーンは、7つの階層にまたがる24の生産ステージからなる有向非巡回グラフ(DAG)としてモデル化されている:

| 階層 | ステージ |
|---|---|
| 入力・IP | 研究/IP、EDAソフトウェア、チップ設計、シリコンウェハー、フォトレジスト、特殊ガス/化学品、ICサブストレート(ABF) |
| 装置 | リソグラフィ(EUV/DUV)、成膜、エッチング・洗浄、CMP・スラリー、計測/検査 |
| 製造 | 先端ファブ(≤7nm)、成熟ノードファブ、メモリファブ(DRAM/NAND) |
| チップ製品 | ロジック・AIアクセラレータ、アナログ/パワー/RF、HBM |
| バックエンド | 先端パッケージング、OSAT/テスト |
| システム | 最終システム(ODM/EMS) |
| エンドマーケット | AIデータセンター、自動車/産業、コンシューマー/モバイル |

各ステージは、サンプルのグローバル価値額($B)、国別市場シェア(合計≤1、残差は未モデル化のロングテール)、そして2つのアナリスト判断スコア(代替可能性、市場感応度)を持つ。

**34本の有向エッジ**がステージ間を接続し(例: `lithography → advanced fab`、`advanced fab → logic/AI`)、伝播エンジンが辿るDAGを構成する。

### 3.2 国(16か国)

米国、中国、台湾、韓国、日本、オランダ、ドイツ、フランス、英国、ベルギー、アイルランド、イスラエル、シンガポール、マレーシア、ベトナム、フィリピン。国別リスクスコアと地図ノードのサイズは**完全に導出値**である——その国が関与する全ステージのシェア加重集計として計算される。手動設定された国スコアは一切ない。

### 3.3 企業(109社)

各企業は、本拠国、生産フットプリントを表す**ステーク**の集合——`{stageId: withinStageShare}`——を持ち、109社のうち一部については**顧客グラフ**および/または**株主テーブル**にもエントリを持つ。(§9の実データ反映の過程でCXMTとKioxiaの2社が追加された。元のデータセットではCXMTのDRAM生産が「YMTC / CXMT」という統合エントリに合算されており、日本のNANDメーカーであるKioxiaは完全に欠落していた。)

### 3.4 顧客グラフ(243の関係)

供給者→顧客の有向データセット: `supplier: [[customer, shareOfSupplierSales], ...]`。これは*収益関係*データセットである——「ASML → TSMC(35%)」は、ASMLの売上の35%がTSMC向けであることを意味し、これはTSMCがリソグラフィ入力についてASMLにどれだけ依存しているかとは異なる量である。この区別が重要になる箇所では、製品は意図的に両方の数値を並べて表示する(§4.5参照)。

逆引きインデックス(`SUPPLIERS`)は、上流起源の検索のために顧客グラフから自動的に導出される。

### 3.5 政策手段(7件)

各政策は深刻度(0〜10)と影響を受けるステージのリストを持つ。例: `{ id: "bis", sev: 9, stages: ["logic_ai","hbm","adv_fab","litho","depo","etch","metro"] }`。ステージごとの政策エクスポージャーはこのテーブルから計算され、手動設定されない。

### 3.6 イベント(42件: 現在ウィンドウのサンプル6件 + 出典付き実在イベント36件)とシナリオ(プリセット3件+カスタム無制限)

イベントは、深刻度、確信度(高/中/低)、経過日数、影響を受けるステージ/国、一次/二次効果のテキスト、背景段落、出典、日付付きタイムラインを持つ。シナリオは、シミュレートされたイベント(深刻度、ステージ、`conf: "Simulated"`、`daysAgo: 0`)を同一のエンジンに注入する——アプリ内シナリオビルダーによるユーザー作成のカスタムシナリオも含む。

イベントテーブルは出所の異なる二層で構成される:

- **現在ウィンドウのサンプル6件** (`e1`–`e6`、`server/src/seed-data.js`) — 凍結スナップショット日付前後に配置された例示用イベント。*現在の*チェーン指数を駆動する。
- **実在の歴史イベント36件** (`h2102_uri` … `h2606_mpban`、`server/src/history-events.js`) — 2021年2月〜2026年6月の実際の出来事(ウィンターストーム・ウリ、ルネサス火災、2022年10月7日のBIS輸出規制、中国のGa/Ge・レアアース規制、Nexperia危機、2025–26年メモリ逼迫など)。各イベントは正準の `dateISO`(そこから `days_ago` をシード/バックフィル時に*導出*)、出典引用、および `app/src/engine/event-assumptions.js` の人手による意味分類を持つ。エンジンの12日半減期によりスナップショット日付までに〜0へ減衰するため、*現在の*指数には寄与しない——複数年の計算履歴(英語版 §4.12)を実データにするために存在する。

### 3.7 株主テーブル(サンプル)

主要企業約25社について、`[ownerName, ownershipShare]` のペアのリストを、公開開示(13Fファイリング、年次報告書)のパターンから作成。国家系資本(政府系ファンド、政府出資、国有企業)にはフラグが付く。

### 4.1 ノードリスクスコア

$$\text{risk} = 0.25\,C_{\text{choke}} + 0.20\,C_{\text{geo}} + 0.20\,C_{\text{policy}} + 0.15\,C_{\text{subst}} + 0.10\,C_{\text{shock}} + 0.10\,C_{\text{market}}$$

6要素中4つは**計算値**であり、2つは**明示されたアナリスト入力**である。UI上のスコア内訳は各要素にその出典タグを付す: `[GRAPH]`、`[HHI]`、`[POLICY DB]`、`[EVENT ENGINE]`、または `[ANALYST]`。

### 4.2 チョークポイント中心性——グラフ構造から計算

DAG上のパス参加中心性: あるノードを通過するsource→sinkの生産経路の数。

$$\text{through}(n) = \text{pathsTo}(n) \times \text{pathsFrom}(n), \qquad C_{\text{choke}}(n) = 10\sqrt{\frac{\text{through}(n)}{\max_m \text{through}(m)}}$$

`pathsTo` と `pathsFrom` は、位相ソート(カーンのアルゴリズム)により $O(V+E)$ で計算される。平方根により分布が圧縮され、中位のノードがゼロ近くに押しつぶされないようにしている。

### 4.3 地理的集中度——ハーフィンダール・ハーシュマン指数(HHI)

各ステージの国別市場シェアに対する標準的なHHI:

$$C_{\text{geo}}(n) = 10\sum_i \text{share}_i^2$$

単一国が独占する場合(例: リソグラフィ: オランダ80%、日本20%)、スコアは $10(0.8^2+0.2^2)=6.8$ となる。

### 4.4 政策エクスポージャー——政策手段データベースより

$$C_{\text{policy}}(n) = \min\left(10,\ \max_p \sigma_p + 0.4\sum_{p \neq \text{max}} \sigma_p\right)$$

$\sigma_p$ はノード $n$ に影響する各政策手段の深刻度。支配的な政策が下限を設定し、副次的な政策は減衰した増分を加える。

### 4.5 ステージ重要度——構造×経済性

$$I_n = 10\left(0.6\cdot\frac{C_{\text{choke}}(n)}{10} + 0.4\cdot\frac{\ln v_n}{\ln v_{\max}}\right)$$

グラフ中心性と対数圧縮された経済価値を組み合わせる(最終組立のような高ボリューム・低差別化のステージが、収益だけで支配的にならないようにするため)。以下の全ての集計指標の重み付けに使用され、フローグラフではノード/ドットのサイズとして視覚的に表現される。

### 4.6 価値加重エッジ

$$w_{a\to b} = \frac{v_b}{\sum_{c \,\in\, \text{out}(a)} v_c}$$

各エッジの重みは、行き先ステージが兄弟ノードの中で占める価値のシェアである。フローグラフではエッジの太さとして表現される。

### 4.7 イベントショック——減衰と伝播

イベント発生ステージにおける発生源ショック:

$$s_0 = \sigma \cdot \kappa_{\text{conf}} \cdot e^{-d/12}$$

$\sigma$ は深刻度(0〜10)、$\kappa_{\text{conf}} \in \{1.0, 0.75, 0.5\}$ は高/中/低の確信度に対応(シミュレートされたイベントは1.0を使用)、$d$ はイベントからの経過日数——おおよそ12日の半減期を与える。

発生源から外向きへの伝播:

$$f_{\downarrow}(a\to b) = 0.55\,(0.5 + 0.5\,w_{a\to b}), \qquad f_{\uparrow} = 0.30$$

下流方向のショックは最大2ホップまで伝わり、各ホップでその特定のエッジの価値重みを用いて $f_\downarrow$ を乗じる(そのため価値の高い経路ほどショックが強く広がる)。上流方向のショックは1ホップのみ、一律 $0.30$ の乗数で伝わる(供給者は下流効果の完全な鏡像ではなく、需要側のこだまを感じる、という考え方)。複数のイベントが同一ノードに影響する場合、そのノードは全ての発生源の中の**最大値**を取る——ショックは単純合算されず、多数のイベントが一つのステージに集中した際の暴走的な値の増大を防ぐ。

### 4.8 チェーン・インパクト指数——ショック場ごとの単一比較可能な数値

$$\text{CII/EII} = \frac{\sum_n s_n \, I_n}{\sum_n I_n}$$

24ノード全体に伝播したショック場の重要度加重平均。イベントのショック場に適用したものが、各イベントカードに表示される**イベント・インパクト指数(EII)**である。企業中断のショック場に適用したものが**企業インパクト指数(CII)**である。

### 4.9 企業インパクト指数——シミュレートされた中断

$$s_0(\text{stage}) = 10 \cdot \text{share}_{c,\text{stage}} \quad \Rightarrow \quad \text{propagate} \quad \Rightarrow \quad \text{CII}_c = \frac{\sum_n s_n I_n}{\sum_n I_n}$$

「企業 $c$ が完全に機能停止したら」をシミュレートするため、エンジンは $c$ が占める全てのステージにショックを注入する。その大きさは域内シェアの10倍とし、確信度1.0、減衰ゼロとする——その上でライブイベントと*全く同一*の伝播関数を実行する。得られたノードごとのショック場をCIIに集計する。企業はこの指数によってランク付けされる。このランキングは数学的な出力であり、恣意的に主張されたリストではない。

### 4.10 企業エクスポージャー——フットプリント加重平均

$$e_{c,\text{stage}} = \text{share}_{c,\text{stage}} \times s_{\text{stage}}, \qquad \bar e_c = \frac{\sum_{\text{stage}} \text{share}_{c,\text{stage}}\, s_{\text{stage}}}{\sum_{\text{stage}} \text{share}_{c,\text{stage}}}$$

*任意の*ショック場(ライブイベント、シナリオ、あるいは他の企業の中断のいずれか)が与えられたとき、これは特定の企業がどれだけ露出しているかを計算する——各イベントの「最も露出している企業」リストと、顧客グラフの伝播ツリーに使用される。

### 4.11 国別スコア——完全な導出値

各国の6つの構成要素は、その国が関与する全ステージの構成要素の**シェア加重平均**であり、これに国タグ付きの直接イベントショック(最大値で合成)を加える。モデル内のどこにも手動設定された国スコアは存在しない。地図レイヤーはステージレベルのデータの純粋関数である。

### 4.12 資本パワー指数(Capital Power Index)

$$\text{CPI}_o = \sum_c \text{own}_{o,c} \cdot \text{CII}_c$$

各所有者 $o$(ファンド、政府機関、ファミリー保有)について、各企業における持株比率をその企業の企業インパクト指数で加重して合計する。これにより「チェーン上の最も構造的に重要な生産能力に対する権利を誰が保有しているか」のランキングが得られる——単純な時価総額加重の持株ランキングとは異なり、企業規模ではなく*チェーン上の重要性*で加重されている点が特徴である。

### 4.13 計算された履歴

$$\text{ChainIndex}(t) = \frac{\sum_n \text{total}(\text{stage}_n, \text{shock}_t)\, I_n}{\sum_n I_n}, \qquad \text{shock}_t = \text{propagate}\big(\{e : e.\text{daysAgo} - t \geq 0\}\big)$$

21日間のスパークラインと「Movers 7D」リストは装飾ではない——各イベントの `daysAgo` を過去にシフトして再伝播させることで、過去の各日について*エンジン全体*を再実行している。これにより、過去の値はライブモデルと内部的に整合しており、別途保守される時系列データではない。

## 6. 技術実装

### 6.1 技術スタック

- **フロントエンド**: React 18 + Vite。`app/` 配下に正式なESモジュール構成のプロジェクトとして構築(`src/components`、`src/data`、`src/engine`、`src/i18n`、`src/utils`)——ブラウザ内でのBabelトランスパイルはもはや行わない。`npm run build` によりミニファイされたバンドル済み成果物が生成される。
- **Leaflet** と **KaTeX** は実際のnpm依存パッケージであり、直接importされる(`import L from 'leaflet'`、`import katex from 'katex'`)。ランタイムにCDNの `<script>` タグから注入する方式ではない。
- **バックエンド**: **SQLite**(`better-sqlite3` 経由)を基盤とする小さなNode/Express API(`server/`)——§3・§9で説明した「vault(金庫)」である。公開の `GET` エンドポイントが現在のデータセットを提供し、ベアラートークンで認証された `/api/admin/*` がライブな書き込み(企業のステーク更新、イベント追加、株主持分の改定など)を、アプリケーションコードの変更やフロントエンドの再デプロイなしに可能にする。
- フロントエンドは起動時に一度だけデータセット全体を取得し(`GET /api/bundle`)、その時点でvaultが保持している内容から派生エンジン(リスクスコア、伝播、ランキング)を構築する。

### 6.2 なぜデータをバンドルの外に出したか

以前のビルドでは、出荷ファイルに焼き込まれた4つの**ハードコードされた**JSテーブル(`STAGES`、`COMPANIES`、`CUSTOMERS`、`POLICIES`)から全てを計算していた——「一つのエンジン、三つの用途」という主張には忠実だったが、データを修正するたびにコード変更と再デプロイが必要という意味でもあった。vaultへの分離は、この関心事をアルゴリズムから切り離す:

- `server/` がデータを所有する——企業、ステージ、顧客グラフ、株主テーブル、政策、イベント、シナリオを、実際のリレーショナル/JSONハイブリッドのSQLiteスキーマとして保持し、主要な修正それぞれについてエビデンス階層の出典を記録する `data_notes` テーブルを備える(§9参照)。
- `app/src/engine/index.js` は、import時に一度だけ計算されるモジュール群ではなく、**ファクトリ関数** `buildEngine(data)` である。vaultが返す内容を受け取り、そこからチョークポイント中心性、HHI、伝播、ランキング、履歴を(再)計算する。v4からの「一つのエンジン、三つの用途」という主張は今も成立している——`propagate()` は依然として一つの関数が三通りに呼ばれる形である——ただし今は静的importに閉じ込められる代わりに、実行時データによってパラメータ化されている。
- データの編集は、今では管理APIを呼ぶこと(あるいは将来の管理UI)であり、JS配列を編集して再ビルドすることではない。新しい企業・政策・イベントを追加すると、以前と同様に依存する全ての数値へ自動的に波及する。導出ロジック自体は変わっておらず、元となるテーブルの取得元が変わっただけである。

トレードオフ: アプリは今やvault APIに到達可能であることに依存する(到達できない場合のためのローディング状態と明確なエラー画面がある——`App.jsx` 参照)。現在の規模(24ステージ、約109社)では、単発の `GET` の後にクライアント側で計算するという方式は依然として1秒未満で完了する単純なものである。1日あたり数千件のイベントを取り込む本番バージョンでは、伝播そのものをサーバー側で計算しキャッシュすることが望ましくなるが、これは `buildEngine` が*どこで*実行されるかの変更であり、アルゴリズム自体の変更ではない。

### 6.3 使用しているグラフアルゴリズム

- **位相ソート**(カーンのアルゴリズム)——パス参加中心性を計算する前に、DAG上の有効な処理順序を確立するために必要
- **前方/後方パスカウント**(`pathsTo`、`pathsFrom`)——位相順序に対する単一パスで実行、$O(V+E)$
- **有界ホップのBFS的伝播**(下流2ホップ、上流1ホップ)——完全なグラフ拡散ではなく、ショック計算を高速に保ち、効果を解釈可能にする(「これは直接効果、これは二次効果」)
- **最大値合成**——複数のショック源を一つのノードに合成する際、合算ではなく最大値を取ることで、無制限な蓄積を防ぐ

### 6.4 描画アプローチ

地図とフローグラフはいずれも(地図はLeaflet API呼び出し、フローグラフは生のSVGで)手作りされており、汎用チャートライブラリを使用していない——これにより、インタラクション(レイヤー横断のタップ・ハイライト)、スタイリング(銅色・端末風の美観)、そして製品固有の視覚的エンコーディング(エッジ幅=価値の重み、ノードサイズ=重要度。既製のチャートタイプでは直接表現できないもの)を完全にコントロールできる。

### 6.5 ロゴシステム

企業識別には、各社の公式ドメインとGoogleの公開ファビコンサービス(`google.com/s2/favicons?domain=...`)を用いる。これは商標化されたアートワークを埋め込んだり再配布したりするのではなく、標準的な指示的使用(nominative use)のパターンであり(多くの金融・ニュースアプリが用いるのと同じ手法)、読み込みに失敗した場合は生成された2文字のモノグラムに縮退する——UIが壊れた画像を表示することはない。

### 6.6 ボールトデータベースの更新

コミットされた SQLite ファイル **`server/data/sscim.db` が全プロダクトデータの唯一の事実源**である。GitHub Pages は静的ホスティングのため Express バックエンドは動かせない——そこでデプロイパイプラインはビルド時にデータベースを JSON スナップショットへ*書き出し*、それを静的サイトにバンドルする。したがってデプロイ済みデータの更新にアプリコードは一切関与しない:

```
sscim.db を編集 → npm run snapshot (app/) → .db をコミット → push → Pages が再ビルド
```

**データベースの編集方法:**

1. **管理 API(単発の編集に推奨)。** ローカルでバックエンドを起動(`cd server && cp .env.example .env`(実際の `ADMIN_TOKEN` を設定)`&& npm ci && npm run dev` → `http://localhost:8787`)し、Bearer トークン認証付きで書き込む。全エンドポイント(`server/src/routes/admin.js`): `PUT/DELETE /api/admin/companies/:id`、`PUT /api/admin/stages/:id`(更新のみ — ステージは DAG の構造ノード)、`PUT/DELETE /api/admin/customers/:supplierId/:customerId`、`PUT/DELETE /api/admin/owners/:companyId/:ownerName`、`POST/PUT/DELETE /api/admin/events[/:id]`、`POST /api/admin/data-notes`。読み取りは各テーブルのミラー + `GET /api/bundle`。
2. **SQL 直接編集**(一括変更): `sqlite3 server/data/sscim.db`(スキーマは `server/src/db.js`)。
3. **キュレーション済み歴史データセット。** 2021–2026年の実在イベントは `server/src/history-events.js` にコードとして存在。編集後は `server/` から `node scripts/backfill-history.mjs` で既存 DB に upsert する。新しいイベント id は `app/src/engine/event-assumptions.js` にも分類を登録すること——未分類 id は表示のみでスコア除外(仕様)。

**公開:** `app/` で `npm run snapshot` を実行 — DB から `vault-snapshot.json` を再生成(ライブ API と同じ `server/src/bundle.js` 経由なのでデータ形状はバイト単位で一致)し、SQLite WAL をチェックポイントして `.db` ファイルを完全な状態にする。`server/data/sscim.db`(+触ったファイル)をコミットして push すると、Pages ワークフロー(`.github/workflows/static.yml`)がサーバー依存をインストールし、コミット済み DB から再書き出し、監査(`npm run audit:data` — 参照切れ・非有限値・範囲外シェアでビルド失敗)、テスト、ビルド、デプロイを行う。

**ブートストラップ/リセット:** 空の DB はサーバー起動時または初回スナップショット時に `seed-data.js` + `history-events.js` から自動シード。`server/` の `npm run seed` が明示的な完全リセット。

**株価(価格 + P/E):** 上場企業(109社中92社; ティッカー対照表は `server/src/tickers.js`)は `quotes` テーブルに市場データ(価格・日中変化・実績/予想 P/E・時価総額)を持ち、企業詳細ビューと COMPANIES フィードに表示される。更新は `server/` から `node scripts/fetch-quotes.mjs`(Yahoo Finance、API キー不要)。GitHub Pages は静的のため「ライブ」とは*最終ビルド時点*を意味する: デプロイワークフローはビルド毎にベストエフォートで再取得し、さらに**毎日の定期実行**(`cron: 0 6 * * *`)により、コミットなしで公開サイトの株価が日次更新される。取得失敗時はコミット済み DB の値にフォールバック。株価は表示メタデータのみで、**いかなるリスク計算の入力にもならない**。

ランディングページ(`index.html`)と紹介ページ(`intro.html`)にも、見出し・セクション・料金の翻訳を含む同等の言語切り替え機能がある。

## 8. エビデンス枠組みとデータの完全性

この方法論は、アルゴリズムの設計と(本番版における)個々のデータポイントの両方を裏付けるために用いる4つのエビデンス区分を定義している:

| 階層 | 説明 | 例 |
|---|---|---|
| **A — 学術的根拠** | 査読済みの基礎理論 | 生産ネットワークにおけるショック伝播(ネットワーク経済学)、ハーフィンダール集中度(産業組織論)、パス中心性(ネットワーク科学) |
| **B — 機関レポート** | 業界・研究機関の刊行物 | SIA/BCGのレジリエンス研究、SEMIの生産能力データ、CSET Supply Chain Explorer、TrendForce/TechInsights/Gartnerのシェア推計 |
| **C — 公的情報源** | 政府・企業の一次資料 | BIS/METI/MOFCOMの規則条文、NISTおよびCHIPSプログラムの文書、EU Chips Act、企業開示資料(10-K/20-F、13F) |
| **D — アナリスト判断** | 明示された専門家の入力 | 代替可能性および市場感応度のスコア。文書化されたルーブリックに照らして採点(Phase 1で正式化予定) |

本番のデータポイントは `[階層 · 出典 · 日付]` を表示するよう設計されている。情報源が競合する場合、想定される挙動は、一つの数値を黙って選ぶのではなく範囲を示すことである。重みと伝播定数(0.55/0.30のホップ係数、12日の減衰定数)は、文書化された過去の事例に照らして較正されるまでは階層D(暫定値)として扱われる(§11参照)。

この枠組みは、元のプロジェクト概要にあるデータ完全性のルールを直接実装したものである: 出典のない主張をしない、あらゆるものに確信度ラベルを付す、推測値ではなく「不明」とする、確定した事実と解釈とを厳格に区別する。

**実データ反映済み、ただしPhase-1の完全なソーシングには未到達。** ベストエフォートの調査(ファブ/パッケージング、装置/材料、メモリ/EDA/設計、アナログ/システム/エンドマーケットの4つの並行調査トラック)により、信頼できる公開情報源——企業の10-K/20-Fファイリング、TrendForce/TechInsights/Gartnerの市場シェア追跡、名前の明記された業界紙——が存在する箇所については、元の例示的なサンプルを現時点の出典付き数値に置き換えた。主要な修正(例: 米国政府によるIntelへの9.9%出資、TSMCの最大顧客がAppleからNvidiaに交代したこと、HBMシェアの修正など)は、vaultの `data_notes` テーブル(`GET /api/data-notes`)に出典とともに記録され、データファイルから相互参照されている。`data_notes` のエントリが**ない**数値は、依然として個別検証されていない引き継ぎのアナリスト判断(§8の階層D)である——このデータセットは意味のある形で改善されたスナップショットではあるが、完全にソーシングされた本番データベースではなく、全ての数値に個別の出典が付くまではそのように表現すべきではない。

**エッジは価値加重であり、生産能力制約ではない。** モデルはステージ間の相対的な価値フローを捉えているが、絶対的な生産能力やボトルネックの飽和はモデル化していない——実際の生産能力制約によるショック(例: ファブの物理的破壊)は、現行の価値重みのみのモデルが予測するのとは異なる形で伝播するだろう。

**伝播定数は未検証の暫定値である。** 下流係数0.55、上流係数0.30、12日の減衰半減期は、データに適合させたものではなく、方向性として妥当な挙動を生むよう選ばれた値である。第11節で較正計画を示す。

**顧客グラフの関係は依存性の一つの軸に過ぎない。** 供給者の顧客への売上シェアは、その顧客の供給者への*入力*依存度と同じではない(ASML → TSMCがASMLの売上の35%であることは、TSMCがASMLに35%依存していることを意味しない——EUVに関して言えば、実際にはほぼ完全な依存に近い)。この混同を防ぐため、製品は関係性の割合とエンジン計算のエクスポージャー数値を意図的に並べて表示しているが、基礎となるデータセットはまだ逆方向(入力側)の集中度を明示的には捉えていない。

**ロングテールの網羅は設計上不完全である。** ステージごとの企業リストは、精選された上位セットと「その他」の残差を示す。約110社という数は意図的な上限であり、これを超えて追加する企業は、チェーン・インパクトのランキングを大きく変えることなくソーシングの負担だけを増やす。

**株主データは急速に陳腐化する。** 持株比率は四半期ごとに変動する。これは製品内で最もコンプライアンス上の配慮を要するデータセットであり、本番運用では最も厳格な出典・日付の管理が求められる。

**クライアント側での計算、サーバー側でのデータ保持。** §6.2で述べた通り、伝播エンジンは単発のバンドル取得の後、依然としてクライアント側で実行される——デモ規模(24ステージ、約109社)では適切だが、1日あたり数千件のイベントを取り込む本番バージョンでは、伝播の計算とキャッシュをサーバー側で行うことが望ましい。

**vaultの可用性が新たな依存関係になった。** ダッシュボードは起動時に `server/` APIに到達可能である必要がある。到達できない場合、アプリは古いデータに黙ってフォールバックするのではなく、明示的なエラー画面を表示する(`App.jsx` の `VaultGate` 参照)。ダッシュボードが機能するには両方のセルフホスティングが必要であり、`app/` 単体の静的デプロイではデータは表示されない。

## 10. ファイル構成とデプロイ

公開される3つのページ——ランディングページ、ガイド、ダッシュボード——はすべて、1つのViteプロジェクト(`app/`)からビルドされるReactページであり、テーマ、i18nのパターン、`Tex`(KaTeX数式レンダリング)などのコンポーネントを共有している。データ(企業、ステージ、顧客グラフ、株主、政策、イベント、シナリオ)は別途、`server/`のvaultに存在する。

```
/
├── README.md / README.ja.md / README.zh.md   本ドキュメント(英語/日本語/簡体字中国語)
├── app/                  サイト本体——Vite + Reactのソース、3つのエントリポイント
│   ├── index.html          Viteエントリ → ランディングページ(ビルド出力: index.html)
│   ├── intro.html          Viteエントリ → ガイドページ(ビルド出力: intro.html)
│   ├── sscim-app.html      Viteエントリ → ダッシュボード(ビルド出力: sscim-app.html)
│   ├── src/
│   │   ├── landing/         Landing.jsx、main.jsx、i18n.js——マーケティング/ポジショニングページ
│   │   ├── intro/            Intro.jsx、main.jsx、i18n.js——ガイド/ウォークスルーページ
│   │   ├── components/       ダッシュボードのUIコンポーネント(Header、FlowGraph、OsmMap、Detail、Briefing、Methodology、Texほか)
│   │   ├── data/              VaultContext.jsx(APIバンドルを取得)+ compMeta.js(静的UIラベル)
│   │   ├── engine/             index.js — buildEngine(data): チョークポイント中心性、HHI、伝播、ランキング、履歴
│   │   ├── i18n/                ダッシュボードの言語辞書 + t()
│   │   ├── utils/                色・ラベルのヘルパー
│   │   ├── App.jsx、main.jsx、theme.js   — ダッシュボードのルート
│   ├── vite.config.js      マルチページビルド(rollupOptions.input: landing/intro/dashboard)、base: './'(相対アセットパス)、出力先は ../dist-app
│   └── package.json
└── server/               vault(金庫)——Node/Express + SQLiteのAPI
    ├── src/
    │   ├── db.js           スキーマ(stages、companies、customers、owners、policies、events、scenarios、data_notes)
    │   ├── seed-data.js     現行のデータセット(実データ反映済みの値。§9参照)
    │   ├── data-notes.js    主要な修正の裏付けとなる出典
    │   ├── seed.js          seed-data.jsからdata/sscim.dbへ投入
    │   ├── routes/public.js  GETエンドポイント群(GET /api/bundle を含む——ダッシュボードの単発起動時取得)
    │   ├── routes/admin.js   ベアラートークン認証による書き込み(PUT/POST/DELETE)
    │   └── index.js          Expressアプリ
    └── package.json
```

**開発環境での起動:** `cd server && npm install && cp .env.example .env && npm run seed && npm run dev`(APIを `:8787` で起動)。続いて `cd app && npm install && npm run dev`(3つのページすべてを `:5173` で起動——`/index.html`、`/intro.html`、`/sscim-app.html`。ダッシュボードは `VITE_API_BASE_URL` を参照し、デフォルトは `http://localhost:8787`)。

**本番デプロイ**では、単一の静的バンドルではなく、二つのものを稼働させる必要がある:
- **フロントエンド**(`app/`): `npm run build` により、ビルド済みの3ページ(`index.html`、`intro.html`、`sscim-app.html`)と共有の `assets/` フォルダを含む `dist-app/` が生成される。このディレクトリをそのまま静的ファイルとしてデプロイする(GitHub Pages、Netlifyなど)。ビルド時に `VITE_API_BASE_URL` をデプロイ済みバックエンドのURLに設定する。
- **バックエンド**(`server/`): 実際のNodeホスト(Render、Fly.io、Railway、VPSなど)が必要である——もはや「HTMLファイルをどこかに置くだけ」のデプロイではない。公開する前に `ADMIN_TOKEN` を実際のシークレット値に設定すること。設定しない場合、管理用書き込みAPIはデフォルトで無効化される(503を返す。オープンにはならない)。

## 11. ロードマップ

段階的計画:

1. **需要検証** — ターゲットユーザー(ファンド、戦略チーム、ジャーナリスト、研究者)との構造化されたデモを実施し、さらなる構築の前に明示的な支払い意思のシグナルを収集する。
2. **ソーシング済みデータベース** — 重要度上位約150〜250ノードについて、サンプル数値を出典付き・日付付きの値に置き換え、代替可能性・市場感応度のアナリスト・ルーブリックを正式化する。
3. **GP Newsブリーフィングの開始** — 手動で精選された内部グラフを用いた週次〜日次のブリーフィングを、需要検証とデータ収集の手段として展開する。
4. **ライブ取り込みパイプライン** — 公的情報源と信頼できるメディアの自動監視、必須の人手レビューを伴うエンティティ抽出とイベント分類、イベント-ノード対応の自動マッピング。
5. **較正** — 文書化された過去の事例(2021年のABF基板不足、2023年のガリウム/ゲルマニウム許認可措置、相次ぐ輸出規制)に照らして伝播パラメータ(ホップ係数、減衰定数、確信度の重み)をバックテストし、いかなるスコアも「サンプルデータ」のラベルなしに出荷される前に行う。
6. **資本フロー・レイヤーの拡張** — 資本レイヤーを持株比率から、有向の資金フローエッジ(CHIPS Act/EU Chips Act/METIの補助金支給、公表されたファブ投資)へと同一グラフ上のトラッキング対象として拡張する。
7. **ダッシュボードのプレミアム階層化** — ブリーフィング購読者が需要を検証した後、完全なインタラクティブ製品(シナリオモード、企業インパクト分析)を有料階層とする。

## 12. 謝辞・方法論の出典

第4節のリスクスコアリング・ショック伝播アルゴリズムは独自のエンジニアリングであるが、ネットワーク科学、産業組織論、生産ネットワークのマクロ経済学における確立された成果の上に構築されている。以下の出典は、各数式が借用している具体的な概念を示すものであり、引用された著者らが本システムを設計・レビュー・推奨しているという主張ではない。*シカゴ・マニュアル・オブ・スタイル*(第17版)著者・年方式に基づいて表記している。

**チョークポイント中心性(§4.2)** は、ネットワーク科学におけるパス参加/媒介性(betweenness)スタイルの中心性を応用している:

> Freeman, Linton C. 1977. "A Set of Measures of Centrality Based on Betweenness." *Sociometry* 40 (1): 35–41. https://doi.org/10.2307/3033543.

**チョークポイント中心性の基盤となる位相ソート(§4.2、§6.3)** は、カーンのアルゴリズムを用いる:

> Kahn, Arthur B. 1962. "Topological Sorting of Large Networks." *Communications of the ACM* 5 (11): 558–62. https://doi.org/10.1145/368996.369025.

**地理的集中度(§4.3)** は、ハーフィンダール・ハーシュマン指数を用いる:

> Hirschman, Albert O. 1945. *National Power and the Structure of Foreign Trade*. Berkeley: University of California Press.
>
> Herfindahl, Orris C. 1950. "Concentration in the U.S. Steel Industry." PhD diss., Columbia University.
>
> Rhoades, Stephen A. 1993. "The Herfindahl-Hirschman Index." *Federal Reserve Bulletin* 79 (3): 188–89.

**ショック伝播、およびショックをネットワーク変動として捉える枠組み(§4.7〜§4.9)** は、生産ネットワークのマクロ経済学に依拠する:

> Acemoglu, Daron, Vasco M. Carvalho, Asuman Ozdaglar, and Alireza Tahbaz-Salehi. 2012. "The Network Origins of Aggregate Fluctuations." *Econometrica* 80 (5): 1977–2016. https://doi.org/10.3982/ECTA9623.
>
> Carvalho, Vasco M. 2014. "From Micro to Macro via Production Networks." *Journal of Economic Perspectives* 28 (4): 23–48. https://doi.org/10.1257/jep.28.4.23.

**複数のショック源を一つのノードで合成する際の最大値合成ルール(§4.7、§6.3)** は、合算ではなく、金融ネットワークの伝染モデルで用いられるカスケード/閾値ロジックに従っている:

> Elliott, Matthew, Benjamin Golub, and Matthew O. Jackson. 2014. "Financial Networks and Contagion." *American Economic Review* 104 (10): 3115–53. https://doi.org/10.1257/aer.104.10.3115.

**エビデンス枠組み(§8)とデータセットの方向性の較正** に情報を与える機関・業界の情報源:

> Semiconductor Industry Association and Boston Consulting Group. 2021. *Strengthening the Global Semiconductor Supply Chain in an Uncertain Era*. Washington, DC: SIA/BCG.
>
> Center for Security and Emerging Technology, Georgetown University. 2021. "The Semiconductor Supply Chain." Issue Brief, January 2021. https://cset.georgetown.edu/publication/the-semiconductor-supply-chain/.
>
> 企業の10-K、20-F、年次報告書の開示情報。SEMIの生産能力統計。TrendForce、TechInsights、Gartnerの市場シェア推計。具体的な数値の引用元となっている箇所は、データファイル内でインラインに出典を示している。

上記のいずれの個人・機関も、SSCIMについて相談を受けたり、レビューしたり、推奨したりしたものではない。これらの出典は方法論的な透明性と学術的な引用慣行のために示すものであり、協力関係の主張ではない。

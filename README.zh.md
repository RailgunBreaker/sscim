## 1. 简介

### 1.1 问题所在

现有的半导体供应链地图——行业联盟图表、研究机构图示、静态信息图——擅长展示**结构**：哪些公司占据链条的哪个环节，哪些国家拥有哪些能力。但它们无法展示**动态**：当新的出口管制规则、地缘政治事件或某家公司的产能决策发生时，哪些节点新近暴露于风险之中，影响会传导多远，以及接下来需要关注什么。

半导体产业的集中度也异常之高。荷兰的一家公司（ASML）几乎生产了全部EUV光刻设备。一个岛屿（台湾）代工了绝大多数最先进制程的逻辑芯片。两家韩国厂商主导着HBM存储器市场。这种集中度意味着冲击——无论是政策性的、地缘政治性的，还是特定公司引发的——会以一种可预测但并不显而易见的方式在链条中传导。

### 1.2 SSCIM是什么

SSCIM是一个实时情报层，整合了：

- 一张**世界地图**（基于真实的OpenStreetMap地理数据），按国家展示半导体能力的分布
- 一张**产业流程图**——展示从研发/IP到终端市场的完整生产链条
- 一个**公司层**——涵盖100多家公司的产能份额、客户关系与股东结构
- 一个**传导引擎**，模拟冲击（事件、假设情景，或单一公司中断）如何在图中扩散
- 一个**情报面板**，解释发生了什么变化、为何重要，以及接下来需要关注什么
- 一个**简报生成器**，根据实时模型状态生成每日情报摘要

一句话描述：*一张实时的半导体供应链情报地图，不仅展示链条身处何方，还展示发生了什么变化、哪些节点受到影响，以及接下来可能发生什么。*

### 1.3 目标用户

关注政策/地缘政治/技术因素如何影响半导体公司的投资者与市场观察人士；研究出口管制与供应链安全的政策与地缘政治风险研究人员；需要快速、结构化了解该行业的学生与分析师；追踪自身风险敞口与关键节点的企业战略团队；以及跑该领域报道的记者。商业假设瞄准这份名单所暗示的中端市场——这些买家无法承担企业级数据供应商的定价，但又需要比免费公开地图更丰富的信息。

### 1.4 配套产品：GP News

SSCIM的引擎可以生成每日半导体情报简报（**GP News**），无需人工从零撰写：应用内的"GP Briefing"功能直接根据实时模型状态，生成"发生了什么变化"、冲击最大的节点、公司风险敞口榜首、国家风险板块以及"接下来需要关注什么"等内容。建议的市场推广顺序是先推出简报——摩擦更小、需求信号反馈更快——再以交互式仪表盘作为高级付费层跟进。

## 2. 产品架构

SSCIM构建为**基于同一计算引擎的三个同步图层**。在任一图层中选择任意元素，都会高亮其他两个图层中的对应元素。

```
┌─────────────────────────────────────────────────────────────┐
│  LAYER 1: WORLD MAP          LAYER 2: INDUSTRY FLOW          │
│  (OpenStreetMap / Leaflet)   (24-stage dependency DAG)       │
│  16 countries, risk-colored  Value-weighted edges,           │
│  derived country links       importance-sized nodes          │
├─────────────────────────────────────────────────────────────┤
│  LAYER 3: INTELLIGENCE PANEL                                 │
│  Event feed · Company impact rank · Movers · Capital rank    │
│  Detail view: score breakdown, spread trees, sources          │
└─────────────────────────────────────────────────────────────┘
                          │
                          ▼
              ┌───────────────────────┐
              │  PROPAGATION ENGINE    │
              │  (one code path, three │
              │   uses: events,        │
              │   scenarios, company   │
              │   disruption)          │
              └───────────────────────┘
```

其核心设计原则——**"一套引擎，三种用途"**——是一项方法论一致性主张：实时事件、假设情景与公司中断模拟并非拥有三套独立公式的三个独立功能，而是同一冲击传导函数的三种不同*输入*。正是这一点，使产品能够可信地宣称自己具备可解释性，而不是把三个黑箱包装在同一个屋檐下。

## 3. 数据模型

### 3.1 阶段（24个节点）

该供应链被建模为一个由24个生产阶段组成、横跨七个层级的有向无环图（DAG）：

| 层级 | 阶段 |
|---|---|
| 输入与IP | 研发/IP、EDA软件、芯片设计、硅晶圆、光刻胶、特种气体/化学品、IC基板（ABF） |
| 设备 | 光刻（EUV/DUV）、沉积、蚀刻与清洗、CMP与抛光液、量测/检测 |
| 制造 | 先进制程晶圆代工（≤7nm）、成熟制程晶圆代工、存储器制造（DRAM/NAND） |
| 芯片产品 | 逻辑与AI加速器、模拟/电源/射频、HBM |
| 后道 | 先进封装、OSAT/测试 |
| 系统 | 终端系统（ODM/EMS） |
| 终端市场 | AI数据中心、汽车/工业、消费/移动 |

每个阶段都带有：一个示例性的全球价值数字（十亿美元），各国市场份额（总和≤1，剩余部分为未建模的长尾），以及两个分析师判断分数（可替代性、市场敏感度）。

**34条有向边**连接各阶段（例如`lithography → advanced fab`、`advanced fab → logic/AI`），构成传导引擎遍历的DAG。

### 3.2 国家（16个）

美国、中国、台湾、韩国、日本、荷兰、德国、法国、英国、比利时、爱尔兰、以色列、新加坡、马来西亚、越南、菲律宾。国家风险分数与地图节点大小均为**完全派生**——通过该国参与的每个阶段的份额加权汇总计算得出。没有任何国家分数是人工设定的。

### 3.3 公司（109家）

每家公司都带有：一个母国，一组**份额（stakes）**——`{stageId: withinStageShare}`，描述其生产布局，以及（109家中部分公司）在**客户图谱**和/或**股东表**中的条目。（CXMT与Kioxia这两家公司是在§9的真实数据核校过程中新增的：原始数据集将CXMT的DRAM产量并入了一个合并条目"YMTC / CXMT"，而日本的NAND制造商Kioxia此前则完全缺失。）

### 3.4 客户图谱（243条关系）

一个有向的供应商→客户数据集：`supplier: [[customer, shareOfSupplierSales], ...]`。这是一个*营收关系*数据集——"ASML → TSMC（35%）"意味着ASML35%的销售额流向TSMC，这与TSMC在光刻设备投入上对ASML的依赖程度是不同的量。在这种区别有意义的地方，产品会刻意将两个数字并排展示（见§4.5）。

一个反向索引（`SUPPLIERS`）由客户图谱自动派生，用于上游来源查询。

### 3.5 政策工具（7项）

每项政策都带有一个严重度（0–10）及受影响阶段列表，例如`{ id: "bis", sev: 9, stages: ["logic_ai","hbm","adv_fab","litho","depo","etch","metro"] }`。每个阶段的政策风险敞口均由该表计算得出，而非人工设定。

### 3.6 事件（6个示例）与情景（3个预设+无限自定义）

事件带有严重度、置信度（高/中/低）、事件天数、受影响的阶段/国家、一阶/二阶影响文本、背景段落、来源引用，以及带日期的时间线。情景则向同一引擎注入一个模拟事件（严重度、阶段、`conf: "Simulated"`、`daysAgo: 0`）——包括用户通过应用内情景构建器自建的自定义情景。

### 3.7 股东表（示例）

针对约25家主要公司，列出一组`[ownerName, ownershipShare]`（股东名称、持股比例）配对，其模式来源于公开披露文件（13F申报、年度报告）。与国家关联的资本（主权基金、政府持股、国有企业）会被标记。

## 4. 风险与影响算法

### 4.1 节点风险分数

$$\text{risk} = 0.25\,C_{\text{choke}} + 0.20\,C_{\text{geo}} + 0.20\,C_{\text{policy}} + 0.15\,C_{\text{subst}} + 0.10\,C_{\text{shock}} + 0.10\,C_{\text{market}}$$

六个分量中有四个是**计算得出**的；两个是**声明的分析师输入**。界面中的每一份分数明细都会为每个分量标注其来源：`[GRAPH]`、`[HHI]`、`[POLICY DB]`、`[EVENT ENGINE]`或`[ANALYST]`。

### 4.2 关键节点中心性——基于图结构计算

DAG上的路径参与中心性：有多少条源→汇生产路径经过某个节点。

$$\text{through}(n) = \text{pathsTo}(n) \times \text{pathsFrom}(n), \qquad C_{\text{choke}}(n) = 10\sqrt{\frac{\text{through}(n)}{\max_m \text{through}(m)}}$$

`pathsTo`与`pathsFrom`通过拓扑排序（Kahn算法）在$O(V+E)$时间内计算得出。平方根用于压缩分布，避免中间层级的节点被压缩至接近零。

### 4.3 地理集中度——赫芬达尔-赫希曼指数（Herfindahl-Hirschman Index）

对每个阶段的国家市场份额计算标准HHI：

$$C_{\text{geo}}(n) = 10\sum_i \text{share}_i^2$$

单一国家垄断的情形（例如光刻：荷兰80%、日本20%）得分为$10(0.8^2+0.2^2)=6.8$。

### 4.4 政策风险敞口——来自政策工具数据库

$$C_{\text{policy}}(n) = \min\left(10,\ \max_p \sigma_p + 0.4\sum_{p \neq \text{max}} \sigma_p\right)$$

其中$\sigma_p$是影响节点$n$的每项政策工具的严重度。主导政策设定下限，次要政策则以衰减后的增量叠加。

### 4.5 阶段重要性——结构×经济

$$I_n = 10\left(0.6\cdot\frac{C_{\text{choke}}(n)}{10} + 0.4\cdot\frac{\ln v_n}{\ln v_{\max}}\right)$$

将图中心性与对数压缩后的经济价值相融合（这样一来，像最终组装这类高产量、低差异化的阶段就不会仅凭营收规模主导指标）。该指标用于对下文所有汇总指数加权，并在流程图中以节点/圆点大小的形式可视化呈现。

### 4.6 价值加权边

$$w_{a\to b} = \frac{v_b}{\sum_{c \,\in\, \text{out}(a)} v_c}$$

每条边的权重是目标阶段在其同级阶段中所占的价值份额，在流程图中以边的粗细呈现。

### 4.7 事件冲击——衰减与传导

事件源头阶段的初始冲击：

$$s_0 = \sigma \cdot \kappa_{\text{conf}} \cdot e^{-d/12}$$

其中$\sigma$为严重度（0–10），$\kappa_{\text{conf}} \in \{1.0, 0.75, 0.5\}$分别对应高/中/低置信度（模拟事件使用1.0），$d$为事件发生以来的天数——由此产生约12天的半衰期。

从源头向外传导：

$$f_{\downarrow}(a\to b) = 0.55\,(0.5 + 0.5\,w_{a\to b}), \qquad f_{\uparrow} = 0.30$$

下游冲击最多传导2跳，每一跳都会使用该条边的具体价值权重乘以$f_\downarrow$（因此冲击沿高价值路径传导得更强）。上游冲击传导1跳，采用固定的$0.30$乘数（供应商感受到的是需求侧的回声，而非下游效应的完全镜像）。当多个事件影响同一节点时，该节点取所有来源中的**最大**冲击值——冲击不会简单相加，从而避免多个事件集中在同一阶段时数值失控。

### 4.8 链条影响指数——每个冲击场对应一个可比较的数字

$$\text{CII/EII} = \frac{\sum_n s_n \, I_n}{\sum_n I_n}$$

对全部24个节点的传导后冲击场取重要性加权平均。将其应用于某一事件的冲击场，即为每张事件卡片上显示的**事件影响指数（Event Impact Index, EII）**。将其应用于某公司中断的冲击场，则为**公司影响指数（Company Impact Index, CII）**。

### 4.9 公司影响指数——模拟中断

$$s_0(\text{stage}) = 10 \cdot \text{share}_{c,\text{stage}} \quad \Rightarrow \quad \text{propagate} \quad \Rightarrow \quad \text{CII}_c = \frac{\sum_n s_n I_n}{\sum_n I_n}$$

为模拟"假如公司$c$被完全中断会怎样"，引擎会在$c$所处的每一个阶段注入一次冲击，其大小为其阶段内生产份额的$10$倍，置信度为1.0且零衰减——然后运行与实时事件*完全相同*的传导函数。由此得到的逐节点冲击场再汇总为CII。公司按此指数排名；该排名是数学计算的输出，而非人为断言的名单。

### 4.10 公司风险敞口——按布局加权的平均值

$$e_{c,\text{stage}} = \text{share}_{c,\text{stage}} \times s_{\text{stage}}, \qquad \bar e_c = \frac{\sum_{\text{stage}} \text{share}_{c,\text{stage}}\, s_{\text{stage}}}{\sum_{\text{stage}} \text{share}_{c,\text{stage}}}$$

给定*任意*冲击场（来自实时事件、情景，或另一家公司的中断），该指标计算特定公司的风险敞口大小——用于每个事件的"风险敞口最大公司"列表，以及客户图谱扩散树。

### 4.11 国家分数——完全派生

每个国家的六个分量，都是该国参与的每个阶段对应分量的**份额加权平均值**，再加上直接标记该国的事件冲击（以最大值方式合并）。模型中没有任何国家分数是人工设定的；地图层是阶段级数据的纯函数。

### 4.12 资本力量指数（Capital Power Index）

$$\text{CPI}_o = \sum_c \text{own}_{o,c} \cdot \text{CII}_c$$

对每个所有者$o$（一支基金、一个政府实体、一个家族控股公司），将其在每家公司的持股比例按该公司的公司影响指数加权后求和。由此得到一份"谁掌握着链条中结构上最重要产能之权益"的排名——这与简单的按市值加权的持股排名不同，因为它是按*链条关键性*加权，而非仅按公司规模。

### 4.13 计算得出的历史

$$\text{ChainIndex}(t) = \frac{\sum_n \text{total}(\text{stage}_n, \text{shock}_t)\, I_n}{\sum_n I_n}, \qquad \text{shock}_t = \text{propagate}\big(\{e : e.\text{daysAgo} - t \geq 0\}\big)$$

21天迷你图与"7日异动（Movers 7D）"列表并非装饰性的——它们通过将每个事件的`daysAgo`向过去平移并重新传导，在过去的每一天重新运行*整个*引擎。这意味着历史数值与实时模型内在一致，而非另外维护的一套独立时间序列。

## 6. 技术实现

### 6.1 技术栈

- **前端**：React 18 + Vite，在`app/`目录下构建为规范的ES模块化项目（`src/components`、`src/data`、`src/engine`、`src/i18n`、`src/utils`）——不再在浏览器内进行Babel转译；`npm run build`会生成经过压缩、打包的构建产物。
- **Leaflet**与**KaTeX**现为真实的npm依赖，直接以import方式引入（`import L from 'leaflet'`、`import katex from 'katex'`），而非在运行时由CDN的`<script>`标签注入。
- **后端**：一个基于**SQLite**（通过`better-sqlite3`）的小型Node/Express API（`server/`）——即§3、§9中所述的"数据金库（vault）"。公开的`GET`接口提供当前数据集；经Bearer令牌认证的`/api/admin/*`接口支持实时写入（更新某公司的份额、新增事件、修订股东持股比例等），且无需改动应用代码或重新部署前端。
- 前端在加载时一次性获取整个数据集（`GET /api/bundle`），并基于金库当前保存的内容构建派生引擎（风险分数、传导计算、排名）。

### 6.2 为何将数据移出代码包

早期版本将全部计算建立在出货文件中**硬编码**的四张JS数据表（`STAGES`、`COMPANIES`、`CUSTOMERS`、`POLICIES`）之上——这忠实体现了"一套引擎，三种用途"的主张，但也意味着每次修正数据都需要改代码并重新部署。将数据金库分离出来，正是为了把这一关注点与算法本身解耦：

- `server/`拥有数据的所有权——公司、阶段、客户图谱、股东表、政策、事件、情景，均保存在一套真实的关系型/JSON混合SQLite模式中，并配有一张`data_notes`表，记录每一项重大修正所对应的证据等级与出处（见§9）。
- `app/src/engine/index.js`现在是一个**工厂函数**`buildEngine(data)`，而不是一组在导入时一次性计算完毕的模块。它接收金库返回的数据，并据此（重新）计算关键节点中心性、HHI、传导、排名与历史记录。v4版本"一套引擎，三种用途"的主张依然成立——`propagate()`依旧是同一个函数，只是以三种方式被调用——只不过现在它的参数来自运行时数据，而不再是封闭在静态import之内。
- 编辑数据现在意味着调用管理API（或未来的管理界面），而不是编辑JS数组再重新构建。新增一家公司、一项政策或一个事件，依然会像以前一样自动波及所有相关联的数字，因为派生逻辑本身没有变化，变化的只是源数据表的获取方式。

代价在于：应用现在依赖于数据金库API的可达性（若无法访问，会显示加载状态及明确的错误页面——见`App.jsx`）。在当前规模下（24个阶段、约109家公司），"一次性`GET`后在客户端计算"的方式依然能在一秒以内完成、足够简单。若生产版本需要处理每日数千个事件，则会希望将传导计算本身迁移至服务端并加以缓存——这只是改变了`buildEngine`*在何处*运行，而非算法本身。

### 6.3 使用的图算法

- **拓扑排序**（Kahn算法），用于在DAG上建立有效的处理顺序，这是计算路径参与中心性之前的必要步骤
- **正向/反向路径计数**（`pathsTo`、`pathsFrom`），通过对拓扑顺序的单次遍历完成——时间复杂度$O(V+E)$
- **有界跳数的类BFS传导**（下游2跳、上游1跳），而非全图扩散，使冲击计算保持快速，且效果可解释("这是直接效应，这是二阶效应")
- 在将多个冲击来源合并到同一节点时采用**最大值合并**而非求和，防止数值无限累积

### 6.4 渲染方式

地图与流程图均为手工构建（地图基于Leaflet API调用；流程图基于原生SVG），而非使用通用图表库——这使产品能够完全掌控交互方式（跨图层点击高亮）、视觉风格（铜色/终端美学），以及产品所需的特定视觉编码（边宽=价值权重，节点大小=重要性，这些都不是任何现成图表类型能够直接表达的）。

### 6.5 徽标系统

公司识别使用各公司的官方域名，配合Google的公共favicon服务（`google.com/s2/favicons?domain=...`），这是一种标准的指示性使用（nominative use）模式（许多金融与新闻类应用都采用同样的方法），而非嵌入或再分发受商标保护的图案。加载失败时会回退为生成的双字母图标——界面永远不会显示损坏的图片。

## 8. 证据框架与数据完整性

该方法论定义了四个证据等级，用于支撑算法设计，以及（在生产版本中）每一个具体数据点的可信度：

| 等级 | 描述 | 示例 |
|---|---|---|
| **A——学术** | 经同行评审的理论基础 | 生产网络冲击传导（网络经济学）、赫芬达尔集中度（产业组织学）、路径中心性（网络科学） |
| **B——机构报告** | 行业与研究机构出版物 | SIA/BCG韧性研究、SEMI产能数据、CSET供应链探索工具（Supply Chain Explorer）、TrendForce/TechInsights/Gartner份额估计 |
| **C——官方来源** | 政府/公司一手文件 | BIS/METI/MOFCOM规则文本、NIST与CHIPS计划文件、欧盟《芯片法案》（EU Chips Act）、公司申报文件（10-K/20-F、13F） |
| **D——分析师判断** | 声明的专家输入 | 可替代性与市场敏感度分数，依据书面评分标准打分（将在第一阶段正式化） |

生产环境中的数据点被设计为显示`[等级 · 引用 · 日期]`。当多个来源存在冲突时，预期行为是展示数值区间，而非悄悄选取其中一个数字。权重与传导常数（0.55/0.30跳数因子、12天衰减常数）在依据有据可查的历史事件完成校准之前（见§11），均被视为D级先验值。

该框架直接落实了原始项目简报中的数据完整性规则：不做无来源的断言、一切都标注置信度、以"不可用"取代臆测数值，以及在已确认事实与解读之间做出严格区分。

## 9. 已知局限性

**已完成一轮真实数据核校，但尚未达到第一阶段的完整数据溯源。** 通过一轮尽力而为的调研（涵盖晶圆代工/封装、设备/材料、存储器/EDA/设计、模拟/系统/终端市场四条并行调研线），凡是存在可靠公开信息源——公司10-K/20-F申报文件、TrendForce/TechInsights/Gartner市场份额追踪，以及具名的行业媒体报道——的地方，均已用当前的、有出处的数字替换了原先的示意性示例数据。重大修正（例如美国政府对Intel持有9.9%股权、Nvidia取代Apple成为TSMC最大客户、HBM份额的修正等）都记录在数据金库的`data_notes`表中（`GET /api/data-notes`），并在数据文件中相互引用。**没有**`data_notes`条目的数字，仍属于沿用下来的、未经逐项核实的分析师判断（§8中的D级）——这份数据集是一次有意义的改进快照，但还不是完全溯源的生产级数据库，在每一个数字都配有独立出处之前，不应将其表述为已完全溯源。

**边是按价值加权的，而非受产能约束的。** 模型捕捉的是各阶段之间的相对价值流动，但并未对绝对生产产能或瓶颈饱和度建模——一次真实的、受产能约束的冲击（例如一座晶圆厂被物理摧毁）传导的方式，会与当前这个仅基于价值权重的模型所预测的不同。

**传导常数是未经验证的先验值。** 0.55下游因子、0.30上游因子与12天衰减半衰期的选取，是为了产生方向上合理的行为，而非拟合数据得出。第11节概述了校准计划。

**客户图谱关系只是依赖性的一个维度。** 供应商流向某客户的销售占比，与该客户在*投入*上对该供应商的依赖程度并不是一回事（ASML → TSMC占ASML销售额的35%，并不意味着TSMC对ASML的依赖程度是35%——具体到EUV，这一依赖接近于完全依赖）。产品刻意将关系百分比与引擎计算的风险敞口数字并排展示，正是为了防止这种混淆，但底层数据集尚未明确捕捉反向（投入侧）的集中度。

**长尾覆盖在设计上是不完整的。** 各阶段的公司列表展示的是精选的头部集合加上"其他"这一剩余项；约110家公司是一个刻意设定的上限——超出这一范围后，增加条目只会增加数据来源负担，而不会实质性地改变链条影响排名。

**股东数据老化很快。** 持股比例按季度变动；这是产品中合规敏感度最高的数据集，在生产环境中需要最严格的引用与日期管理规范。

**客户端计算，服务端数据。** 如§6.2所述，传导引擎在单次获取数据包之后，依然在客户端运行——这在演示规模下（24个阶段、约109家公司）是合适的，但在需要处理每日数千个事件的生产版本中，会希望将传导计算与缓存放在服务端进行。

**数据金库的可用性现在是一项依赖。** 仪表盘在加载时需要能够访问`server/`API。若无法访问，应用会显示明确的错误页面，而不是悄悄回退到陈旧数据（见`App.jsx`中的`VaultGate`）。仪表盘要正常运作，需要将前后端两部分都自行托管——仅静态部署`app/`本身不会显示任何数据。

**开发环境启动方式：** `cd server && npm install && cp .env.example .env && npm run seed && npm run dev`（在`:8787`端口启动API），然后 `cd app && npm install && npm run dev`（在`:5173`端口启动仪表盘，读取`VITE_API_BASE_URL`——默认值为`http://localhost:8787`）。

**生产环境部署**现在需要同时运行两个部分，而不再是单一的静态包：
- **前端**（`app/`）：`npm run build`会生成`dist-app/`；将其与根目录的`index.html`/`intro.html`一起作为静态文件部署（GitHub Pages、Netlify等）。构建时需将`VITE_API_BASE_URL`设置为已部署后端的URL。
- **后端**（`server/`）：需要一个真正的Node托管环境（Render、Fly.io、Railway、VPS等）——不再是"把HTML文件放到任何地方"式的部署。在对外公开之前，务必将`ADMIN_TOKEN`设置为真实的密钥值。若不设置，管理写入API默认会被禁用（返回503，而不是开放状态）。

## 11. 路线图

分阶段计划：

1. **需求验证**——面向目标用户（基金、战略团队、记者、研究人员）开展结构化演示；在进一步构建之前收集明确的付费意愿信号。
2. **有据可查的数据库**——为约150–250个最重要的节点，用有引用、有日期的数值取代示例数字；将可替代性与市场敏感度的分析师评分标准正式化。
3. **GP News简报上线**——以每周至每日的简报作为需求验证与数据收集的载体，使用人工整理的内部图谱。
4. **实时数据接入管线**——自动监测官方来源与可信媒体，进行实体提取与事件分类并强制人工复核，自动完成事件到节点的映射。
5. **校准**——在任何分数摆脱"示例数据"标签之前，先针对有据可查的历史事件（2021年ABF基板短缺、2023年镓/锗出口许可行动、历次出口管制升级）对传导参数（跳数因子、衰减常数、置信度权重）进行回测。
6. **资本流动层扩展**——将资本层从持股比例扩展至有向的资金流动边：将CHIPS Act/EU Chips Act/METI的补贴发放及已公布的晶圆厂投资，作为同一图谱上被追踪的资金流。
7. **仪表盘高级付费层**——一旦简报订阅用户验证了需求，即推出完整的交互式产品（情景模式、公司影响分析）作为付费层级。

> Kahn, Arthur B. 1962. "Topological Sorting of Large Networks." *Communications of the ACM* 5 (11): 558–62. https://doi.org/10.1145/368996.369025.
